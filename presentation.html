Data Flow

Request Flow & Event Processing

HTTP Request Flow (Service Consumption)

  1. Client ‚Üí Sends request to API Gateway (port 8080)
  2. API Gateway ‚Üí Routes request to Load Balancer (no auth required for consumption)
  3. Load Balancer ‚Üí Queries Service Discovery for healthy containers
  4. Service Discovery ‚Üí Returns list of containers from Consul cache
  5. Load Balancer ‚Üí Selects container using Round Robin and routes request
  6. Container ‚Üí Processes request and returns response
  7. Response ‚Üí Flows back through reverse path to client

Note: JWT authentication is only required for management operations (create/update/delete containers and images), not for consuming deployed services.

Kafka Event Flow

  1. User ‚Üí Creates Docker image via UI and uploads to Orchestrator
  2. Orchestrator ‚Üí Builds image, creates/stops/deletes containers
  3. Orchestrator ‚Üí Publishes event to "container-lifecycle" topic
  4. Service Discovery ‚Üí Consumes event and registers in Consul
  5. Billing Service ‚Üí Consumes event and calculates costs
  6. Consul ‚Üí Executes TCP health checks on containers
  7. Service Discovery ‚Üí Updates cache with healthy/unhealthy status

Resilience & Fault Tolerance

Fault Tolerance Mechanisms

Circuit Breaker

States:

  • Closed: Normal operation
  • Open: After 3 consecutive failures
  • Half-Open: Retry after 15s

Benefits:

  • Prevents cascading failures
  • Fail fast on issues
  • Automatic self-recovery

Fallback Cache

Features:

  • Cache of last successful response
  • 10 second TTL
  • Per app_hostname

Benefits:

  • Continuous routing during failures
  • Graceful degradation
  • Better user experience

Health Checks

Service Health Checks

  • /health endpoint on each service
  • Check every 30 seconds
  • 10 second timeout
  • 3 retries before marking unhealthy

Container Health Checks

  • TCP health checks via Consul
  • Check every 10 seconds
  • Auto-deregistration if unhealthy
  • Routing only to healthy containers

Technical Challenges

Key Implementation Challenges

1. Docker-in-Docker (DinD) - Networking Challenges

Problem: The Orchestrator needs to create and manage containers dynamically from within a container. Docker doesn't allow running the Docker daemon inside a container natively.

Solution: Used Docker-in-Docker container (`docker:24-dind`) that runs the Docker daemon. Orchestrator connects via DOCKER_HOST=tcp://docker-dind:2375.

Challenge: Containers are created inside docker-dind, which has its own network context. The nvidia-network exists on the Docker Compose host, not inside docker-dind. Solution: Expose ports dynamically and access containers through docker-dind:external_port.

2. Dynamic Port Mapping

Problem: Cannot assign fixed ports because they would collide when multiple containers are created.

Solution: Use ports={'80/tcp': None} for dynamic port assignment, then read the assigned port after container creation from container.attrs['NetworkSettings']['Ports'].

4. Circuit Breaker & Fallback Cache

Problem: Prevent cascading failures if Service Discovery fails. Need to keep routing even when downstream services are temporarily unavailable.

Solution: Circuit Breaker pattern - opens after 3 consecutive failures, retries after 15s (half-open state), auto-closes on success. Fallback cache maintains last successful response per app_hostname for 10 seconds, allowing graceful degradation.

5. Dynamic Load Balancing

Problem: Route requests to containers that can appear/disappear dynamically. Need real-time service discovery integration.

Solution: Round Robin algorithm over healthy container list obtained from Service Discovery. Async ServiceDiscoveryClient with in-memory cache. Fallback cache ensures routing continues even if Service Discovery is temporarily down.

6. Real-time Billing

Problem: Calculate costs in real-time based on container lifecycle events. Need to track usage accurately as containers are created, started, and stopped.

Solution: Event-driven architecture - Billing Service consumes Kafka events from "container-lifecycle" topic. Calculates costs per minute ($0.01/min per container) and updates PostgreSQL records in real-time. Handles container.created, container.started, and container.stopped events.
